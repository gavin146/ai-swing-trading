# SwingFi Copilot Observability Specification

This spec defines the production signals SwingFi needs before Copilot becomes a multi-user beta. It avoids storing raw portfolio payloads, broker credentials, provider tokens, or OpenAI prompts containing full account data.

## Event Principles

- Log operational metadata, not portfolio payloads.
- Every event should include `source`, `status`, `started_at` or `created_at`, and a sanitized error when relevant.
- Every provider datum must track both `fetched_at` and `data_as_of`.
- User-level cost metadata may use user id, report id, input hash, model, token estimate, and status, but must not include full holdings, account numbers, tokens, or raw prompts.
- Admin alerts should be useful without exposing secrets or full customer financial context.

## Sync Run Status

Track each sync attempt with:

- `sync_run_id`
- `user_id`
- `provider_id`
- `connection_id`
- `status`: `running`, `completed`, `partial`, `failed`
- `started_at`
- `completed_at`
- `account_count`
- `position_count`
- `data_as_of`
- `sanitized_error_code`
- `sanitized_error_message`
- `retryable`

Existing schema foundation: `portfolio_sync_runs` in `/Users/gavin/Documents/ai swing trading/db/copilot-schema-migration.sql`.

## Provider Health

Track:

- provider id
- enabled/disabled state
- last successful sync
- last health check
- reconnect-required state
- rate-limit state when supplied by provider
- sanitized provider error count by day

Do not log provider credentials, full external account ids, access tokens, refresh tokens, or raw provider responses.

Existing schema foundation: `brokerage_connections`.

## Source Freshness

Track per report and per position:

- quote freshness
- portfolio snapshot freshness
- SwingFi opportunity freshness
- news freshness
- filing freshness
- earnings/event freshness
- macro/regime freshness
- calibration freshness

Recommended status values:

- `fresh`
- `stale`
- `missing`
- `error`

## Report Generation Status

Track:

- `copilot_report_id`
- `user_id`
- `report_date`
- `input_hash`
- `generation_status`: `queued`, `running`, `completed`, `partial`, `failed`
- deterministic builder version
- narrator id
- prompt version
- model
- output status
- sanitized error
- created/completed timestamps

Existing schema foundation: `copilot_reports`.

## Narrator Fallback Rate

Track daily:

- total narration attempts
- deterministic-only count
- OpenAI success count
- timeout fallback count
- validation fallback count
- provider error fallback count
- cache hit count
- average latency
- estimated cost

This should be visible in admin before OpenAI narration is enabled for multi-user Copilot.

## Paper Risk Policy Metrics

Track only when paper simulation is explicitly enabled:

- total order intents reviewed
- approved count
- rejected count
- rejection count by `RiskPolicyRejectCode`
- duplicate idempotency attempts
- stale quote rejections
- defensive-regime rejections
- earnings-blackout rejections
- max exposure rejections
- no LLM override attempts accepted

Do not store customer-facing direct-trade instructions. Paper events must say simulation only.

## Duplicate-Order Attempts

Track:

- idempotency key hash, not raw payload
- user id
- signal/opportunity reference
- first seen timestamp
- duplicate timestamp
- rejection code

This helps detect retry bugs without logging full portfolio state.

## Admin Alerts

Send admin alerts for:

- provider sync failure spike
- no Copilot report generated by expected time
- OpenAI fallback rate above threshold
- daily ranking scan blocked or too thin
- Supabase persistence failure
- email delivery failure spike
- repeated duplicate paper-order attempts
- policy rejection spike by reason

Admin emails should include counts, source ids, run ids, and sanitized errors only.

## Per-User Cost Metadata

Track:

- user id
- feature area: `copilot_report`, `assistant_chat`, `opportunity_explanation`
- input hash
- model
- prompt version
- estimated input/output tokens
- provider status
- cache status
- created_at

Do not store raw prompts with full portfolio payloads unless a separate retention, consent, and redaction policy is approved.

## Minimum Admin Dashboard Cards

Before Copilot beta:

- latest sync run
- provider health
- report generation status
- narrator fallback rate
- stale source count
- paper risk rejection count
- duplicate-order count
- admin alert count
- per-user estimated cost summary
